# Splynek localization

Splynek ships in **six languages** as of v1.6.1:

| Locale | Language | Status |
|--------|----------|--------|
| `en`   | English (source) | Always present |
| `pt-PT` | Portuguese (Portugal) | 118/118 strings (100%) |
| `es`   | Spanish | 118/118 strings (100%) |
| `fr`   | French | 118/118 strings (100%) |
| `de`   | German | 118/118 strings (100%) |
| `it`   | Italian | 118/118 strings (100%) |

> **Note**: `pt-PT` is European Portuguese. Brazilian Portuguese (`pt-BR`) speakers will fall through to whatever Apple's locale-resolution prefers — typically English. Adding `pt-BR` is straightforward (see below).

## Source of truth

The source of truth is **`Scripts/regenerate-localizations.py`**, a Python file containing every string + every translation as readable Python dicts.

**Why a generator script instead of editing the catalog directly:**
- The catalog (`Sources/SplynekCore/Localizable.xcstrings`) is JSON wire format. 459 lines for 56 strings. Hand-editing it for new strings or new languages is unreviewable.
- The Python file groups translations by feature (onboarding, Agents tab, Downloads, etc.), which makes drive-by improvements easy.
- AI-assisted translations all live in one inspectable place.
- `git diff` on a translation update shows the actual change in plain English/Portuguese/etc., not 50 lines of nested JSON.

## Workflow

### 1. Add a new English string to a view

```swift
// SwiftUI
Text("Some new label")           // ← already a LocalizedStringKey
Label("Run audit", systemImage: "...")

// Programmatic
NSLocalizedString("Some new label", comment: "")
```

That's it from the code side. SwiftUI's `Text(_: StringLiteralType)` initializer takes a `LocalizedStringKey`, so the string literal IS the catalog key — no separate registration.

### 2. Add the string to the catalog

Edit `Scripts/regenerate-localizations.py`. Add an entry under `NEW_V16_STRINGS` (or rename to `NEW_VN_STRINGS` for future releases):

```python
NEW_V16_STRINGS = {
    # … existing entries …
    "Some new label": {
        "pt-PT": "Alguma nova etiqueta",
        "es":    "Alguna etiqueta nueva",
        "fr":    "Un nouveau libellé",
        "de":    "Ein neues Label",
        "it":    "Una nuova etichetta",
    },
}
```

### 3. Regenerate the catalog

```bash
python3 Scripts/regenerate-localizations.py
```

Output looks like:

```
✓ wrote 119 strings across locales: ['de', 'es', 'fr', 'it', 'pt-PT']
  de:    119/119 (100%)
  es:    119/119 (100%)
  fr:    119/119 (100%)
  it:    119/119 (100%)
  pt-PT: 119/119 (100%)
```

If a translation is missing for any locale, the script reports the gap and that locale's coverage drops below 100%.

### 4. Build + verify

```bash
swift run splynek-test         # 166 tests, all pass
swift build --target SplynekCore
./Scripts/build.sh debug
open build/Splynek.app
```

To verify Portuguese specifically: switch the system to Portuguese (System Settings → Language & Region → Português (Portugal)) and re-launch Splynek. Or use NSLocalizationLogVerbose for runtime debugging:

```bash
defaults write app.splynek.Splynek AppleLanguages -array "pt-PT"
defaults write app.splynek.Splynek NSLocalizationLogVerbose -bool YES
open build/Splynek.app
```

### 5. Commit both

Always commit `Scripts/regenerate-localizations.py` AND `Sources/SplynekCore/Localizable.xcstrings` together. The script is the source of truth; the catalog is the build artefact. Drift between them = lost translations.

## Adding a new locale

To ship a new language (e.g. `pt-BR`, `nl`, `ja`):

1. Add the locale to `LOCALES` in `Scripts/regenerate-localizations.py`.
2. Provide `pt-BR` (or whichever) translations for **every** entry in `NEW_V16_STRINGS`. The 56 original strings will need separate handling — extend `PT_PT_FOR_EXISTING` with a parallel `PT_BR_FOR_EXISTING`.
3. Add the locale to `CFBundleLocalizations` in:
   - `Resources/Info.plist` (DMG / SPM build)
   - `project.yml` MAS target (`CFBundleLocalizations: [en, pt-PT, …]`)
4. `python3 Scripts/regenerate-localizations.py`
5. Verify all coverage is at 100%.
6. Apple's Mac App Store auto-updates the listing's "Languages" section from `CFBundleLocalizations` after the next archive uploads.

## Translation quality notes

All translations in v1.6.1 are AI-generated by Anthropic Claude (Opus 4.7) reviewed by the original maintainer. They follow Apple's macOS localization conventions:

- **`pt-PT`**: European Portuguese conventions — "ficheiro" not "arquivo", "ecrã" not "tela", "telemóvel" not "celular". Imperative tense for buttons, formal-you ("Pode adicionar…") for prose. Apple's macOS Portuguese style.
- **`es`**: Castilian Spanish, modern Apple "tu" form in prose, imperative for buttons.
- **`fr`**: Standard French with Apple style.
- **`de`**: Standard German with Apple's "Sie" form.
- **`it`**: Italian with Apple's "tu" form.

**Native-speaker review is welcomed.** PRs against `Scripts/regenerate-localizations.py` are easy to review — each translation lives next to its English source on one line. If you spot a translation that reads as machine-generated, please open an issue or PR.

## Where translations are NOT applied (yet)

Some surfaces still ship in English on every locale:

- The **Sovereignty + Trust catalog data** (~1,155 sovereignty entries, ~30 trust entries). The `note` strings on each alternative would need to translate ~5,000 short strings. Out of scope for v1.6.1.
- The **README.md** + **CHANGELOG.md** + **CONTRIBUTING.md** + **MCP_SETUP.md**.
- Marketing materials (LANDING.md, SHOW_HN.md, PRESS_KIT.md).
- The **CLI** (`splynek-cli`).
- **OpenAPI** descriptions exposed via `/splynek/v1/openapi`.

The user-visible app UI (sidebar, tabs, onboarding, Settings, Agents tab, Downloads, Sovereignty, Trust, alerts) is fully localized. That covers > 95% of what a non-English-speaking user actually reads.
